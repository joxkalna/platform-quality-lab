# Pact: The Big Picture

## What is Pact?

Pact is a contract testing framework that ensures services can communicate with each other. Instead of spinning up all services for integration tests, Pact lets each service verify its API contracts independently.

## The 4 Components

### 1. Consumer

The service that makes the HTTP request (the client).

- Defines **expectations** of what it needs from a provider
- Generates a **pact file** (JSON contract) from its tests
- Publishes the pact file to the Pact Broker

### 2. Provider

The service that responds to the HTTP request (the API).

- Replays the consumer's expectations against its real API
- Reports verification results back to the Pact Broker
- Can verify against multiple consumers simultaneously

### 3. Pact Broker

The central hub that stores and manages contracts.

- Stores pact files published by consumers
- Stores verification results from providers
- Provides a dependency graph (who talks to whom)
- Enables the **can-i-deploy** check before releases
- Exposes a UI for browsing contracts and their status

### 4. CI/CD Pipeline

The glue that automates the workflow.

- Consumer pipeline: run tests → generate pact → publish to broker
- Provider pipeline: fetch pacts from broker → verify → publish results
- Deployment gate: `can-i-deploy` blocks releases if contracts are broken

## How It All Fits Together


## Key Concepts

### Pact File

A JSON contract generated by consumer tests. Contains:
- The consumer and provider names
- A list of interactions (request/response pairs)
- Metadata (pact specification version)

### Verification

The provider replays each interaction from the pact file against its running API. If all responses match, the contract is verified.

### Can-I-Deploy

A CLI check that queries the Pact Broker:
> "Is it safe to deploy version X of service Y to environment Z?"

It answers this by checking whether all contracts involving that service version have been successfully verified.

### Tags & Environments

- **Tags** label a participant version (e.g. `main`, `feat-xyz`)
- **Environments** (Pact Broker 2.80+) model deployment targets (e.g. `staging`, `production`)
- `record-deployment` track what's actually running where - most likely just use this. 
- `record-release` - use only when multiple version are required and co-exist in the same env: mobile app, npm package, cli tool etc

## Typical Workflow

1. Consumer writes a test describing what it expects from the provider
2. Pact framework generates a pact file from that test
3. Consumer CI publishes the pact file to the Pact Broker
4. Provider CI fetches pacts for its consumers from the Broker
5. Provider verifies each interaction against its real API
6. Verification results are published back to the Broker
7. Before deploying either service, `can-i-deploy` checks compatibility
8. If all contracts are green, deploy proceeds

## Infrastructure Overview

A typical Pact Broker deployment consists of:

- A **container** running the Pact Broker application (e.g. `pactfoundation/pact-broker` from Docker Hub)
- A **Postgres database** as the backing store (recommended by Pact)
- A **load balancer** in front of the container
- **DNS** pointing to the load balancer
- **Basic auth** with separate read-only and read-write credentials
- **Secrets store** for credential storage

For local development, this project runs the Broker on a Kind cluster — see [04-broker-ops.md](./04-broker-ops.md#local-kind-deployment) for setup steps.

## Monorepo vs Multi-Repo

This project keeps everything in one repo for simplicity. In a real organisation, the broker infrastructure, provider initialisation, shared verification library, and each service would live in separate repos owned by different teams.

See [06-repo-separation.md](./06-repo-separation.md) for the full mapping — what each repo contains, how they connect, and a checklist for scaling from monorepo to multi-repo.
